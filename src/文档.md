# 娱乐大厅 WebSocket 消息协议草案

> 适用对象：Vue 前端 / 后端实时消息实现 / 联调同学
>
> 目标：
> 1. 给前端一个统一的实时消息格式
> 2. 当前后端已按该协议实现 P1 基础版，前端可直接联调或继续按它做 Mock
> 3. 后端后续增强时尽量保持协议不改，减少前端重构

---

## 2. 协议设计原则

1. 所有消息都使用统一 envelope
2. 消息必须带 `scene`
3. 消息必须带 `event`
4. 消息必须能通过 `roomId/gameId/gameCode` 定位上下文
5. 前端所有实时状态更新都通过 `payload` 处理
6. 服务端错误也走统一消息结构

---

## 3. WebSocket 连接建议

### 3.1 建议连接地址
```text
/ws
```

### 3.2 鉴权方式建议
任选一种：

#### 方案 A：连接时 query 传 token
```text
ws://host/ws?token=xxxxx
```

#### 方案 B：先连接，再发送认证消息
```json
{
  "scene": "system",
  "event": "AUTH",
  "traceId": "uuid",
  "timestamp": 1710000000000,
  "payload": {
    "token": "xxxxx"
  }
}
```

推荐：**方案 B**，更清晰。

---

## 4. 统一消息包结构

所有消息统一使用如下格式：

```json
{
  "scene": "room",
  "event": "PLAYER_READY",
  "traceId": "9f41c39d-64c3-4af0-8d95-49d0c7c4f1f4",
  "timestamp": 1710000000000,
  "success": true,
  "message": "ok",
  "roomId": "room_1001",
  "gameId": "game_2001",
  "gameCode": "SHANDONG_MAHJONG",
  "payload": {}
}
```

---

## 5. 顶层字段说明

| 字段 | 类型 | 必填 | 说明 |
|---|---|---:|---|
| `scene` | string | 是 | 场景：`system` / `lobby` / `room` / `mahjong` / `gouji` |
| `event` | string | 是 | 事件名称 |
| `traceId` | string | 是 | 消息追踪 ID |
| `timestamp` | number | 是 | 服务端时间戳 |
| `success` | boolean | 是 | 是否成功 |
| `message` | string | 否 | 附加提示信息 |
| `roomId` | string | 否 | 房间 ID |
| `gameId` | string | 否 | 对局 ID |
| `gameCode` | string | 否 | `SHANDONG_MAHJONG` / `SHANDONG_GOUJI` |
| `payload` | object | 是 | 具体消息体 |

## 6. scene 划分

### `system`
系统级消息：
- 鉴权
- 心跳
- 错误
- 重连恢复

### `lobby`
大厅级消息：
- 游戏列表变化
- 房间列表变化
- 在线人数变化

### `room`
房间级消息：
- 玩家加入
- 玩家离开
- 玩家准备
- 开始游戏
- 房间解散

### `mahjong`
麻将对局消息：
- 发牌
- 出牌
- 摸牌
- 碰/吃/杠/胡
- 结算

### `gouji`
够级对局消息：
- 发牌
- 出牌
- 过牌
- 清轮
- 回合结束
- 进贡
- 还贡
- 结算

## 7. 系统消息协议

---

### 7.1 鉴权请求
```json
{
  "scene": "system",
  "event": "AUTH",
  "traceId": "uuid",
  "timestamp": 1710000000000,
  "payload": {
    "token": "xxxxx"
  }
}
```

### 7.2 鉴权响应
```json
{
  "scene": "system",
  "event": "AUTH_SUCCESS",
  "traceId": "uuid",
  "timestamp": 1710000000000,
  "success": true,
  "message": "auth success",
  "payload": {
    "userId": "u1001",
    "nickname": "玩家A"
  }
}
```

失败时：
```json
{
  "scene": "system",
  "event": "AUTH_FAILED",
  "traceId": "uuid",
  "timestamp": 1710000000000,
  "success": false,
  "message": "token invalid",
  "payload": {}
}
```

---

### 7.3 心跳

```json
{
  "scene": "system",
  "event": "PING",
  "traceId": "uuid",
```

```json

  "scene": "system",

  "traceId": "uuid",
  "timestamp": 1710000000001,
```

## 12. 错误消息统一格式

无论哪个 scene，服务端都建议统一发：

```json
{
  "scene": "gouji",
  "event": "ERROR",
  "traceId": "uuid",
  "timestamp": 1710000000000,
  "success": false,
  "message": "当前出牌无法压过上一手牌",
  "roomId": "room_2001",
  "gameId": "game_3001",
  "payload": {
    "code": "PLAY_INVALID"
  }
}
```

建议错误码：
- `AUTH_FAILED`
- `ROOM_NOT_FOUND`
- `ROOM_FULL`
- `GAME_NOT_FOUND`
- `TURN_INVALID`
- `PLAY_INVALID`
- `TRIBUTE_INVALID`
- `RETURN_TRIBUTE_INVALID`
- `GAME_PHASE_INVALID`
- `SYSTEM_ERROR`

## 13. 前端落地建议

### Vue 前端 socket 层建议
建议封装为：

```ts
interface SocketEnvelope<T = any> {
  scene: 'system' | 'lobby' | 'room' | 'mahjong' | 'gouji'
  event: string
  traceId: string
  timestamp: number
  success: boolean
  message?: string
  roomId?: string
  gameId?: string
  gameCode?: 'SHANDONG_MAHJONG' | 'SHANDONG_GOUJI'
  payload: T
}
```

### 消息处理建议
按 `scene + event` 分发：
- `lobbyStore`
- `roomStore`
- `mahjongGameStore`
- `goujiGameStore`

### 当前前端最适合做的
前端现在可以：
- 先按本文协议写消息类型定义
- 直接对接当前 `/ws` + `/topic/**` 基础版实现
- 直接对接当前 `/app/system/auth`、`/app/system/ping` 与 `/user/queue/system`
- 对未实现的重连恢复部分继续用 Mock 推动界面

---

## 14. 最终建议

### 先做什么
1. 前端按本文把 socket 协议和类型写好
2. 前端优先接入 `/topic/lobby/rooms`、`/topic/room/{roomId}`、`/topic/game/{gameId}`
3. 接入 `/app/system/auth`、`/app/system/ping` 与 `/user/queue/system`
4. 对重连恢复继续使用 Mock 方案

### 当前不要做什么
- 不要现在就绑定尚未实现的重连恢复协议细节
- 不要假设已经有观战同步和全量恢复推送

---

## 15. 一句话总结

> 这份协议是给前端“先开发、后联调”用的：
> 前端现在就可以按这套 envelope 和事件名把大厅、房间、麻将、够级的实时状态流全部搭起来，
> 当前已落地的 P1 基础版会尽量持续保持与本文协议一致。
